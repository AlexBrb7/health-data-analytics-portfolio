Predicción de conversión en e-commerce parafarmacéutico (Google Analytics)

por Alejandro Sanz Serrano

Farmacéutico + Máster en Data Cience & Business Analytics

En el entorno actual del comercio electrónico, uno de los principales retos para las empresas del sector parafarmacéutico es identificar qué usuarios presentan una mayor probabilidad de conversión durante su navegación en la web. La optimización de este proceso permite mejorar la eficiencia de las campañas de marketing, personalizar la experiencia del usuario y maximizar el retorno de la inversión.

Este proyecto aborda este problema mediante el análisis de datos de comportamiento web basados en métricas de Google Analytics, representativos de un entorno de e-commerce.

El objetivo principal es desarrollar un modelo predictivo capaz de estimar la probabilidad de conversión de un usuario durante una sesión, con el fin de:

  1. Identificar usuarios con alta intención de compra
  2. Priorizar acciones comerciales y de marketing
  3. Optimizar la asignación de recursos
  4. Mejorar la tasa de conversión

Datos: El análisis se realiza sobre un dataset de una parafarmacia anónima que realiza venta online de comportamiento de usuarios en entornos de e-commerce, basado en variables típicas de Google Analytics (tiempo de navegación, número de páginas visitadas, tasa de rebote, etc.).

Enfoque metodológico: El proyecto sigue un pipeline completo de análisis de datos:

  1. Análisis exploratorio de datos (EDA)
  2. Preprocesamiento y transformación de variables
  3. Modelización predictiva (Regresión Logística, Random Forest, Redes Neuronales)
  4. Evaluación mediante métricas robustas (ROC-AUC)
  5. Optimización de hiperparámetros
  6. Interpretación de resultados y análisis de variables clave
  7. Aplicación práctica a negocio
  8. Valor aportado

El modelo desarrollado permite pasar de un enfoque reactivo a uno proactivo, facilitando la identificación de usuarios con mayor probabilidad de conversión en tiempo real. Esto abre la puerta a múltiples aplicaciones en un entorno empresarial, como la segmentación avanzada de clientes, la personalización de la experiencia de usuario o la optimización de campañas de marketing.

Consideraciones: Se incluyen limitaciones relevantes del modelo, como la ausencia de identificadores únicos de usuario o la falta de información temporal completa, lo que refuerza el carácter realista y crítico del análisis realizado.

Será importante analizar con más detalle su definición antes de utilizarla en el modelo. Si PageValues está calculado utilizando información posterior a la conversión, introduciría data leakage (es decir, el valor es alto cuando alguien compra, el modelo está "haciendo trampa"). En ese caso, lo más prudente sería excluir esta variable del modelo o analizar su definición en detalle antes de utilizarla. En este caso, en vez de eliminar la variable, se va a probar el modelo con y sin ella y, si el modelo depende demasiado de PageValues, podría no generalizar bien y es mejor eliminarla.

# Aplicar one-hot encoding
df_encoded = pd.get_dummies(df, drop_first=True)
Apunte: se utiliza drop_first=True para evitar la multicolinealidad derivada del llamado dummy variable trap. Si se incluyeran todas las categorías, una de ellas podría expresarse como combinación lineal de las demás. Por ello, se elimina una categoría de referencia, de modo que el resto de variables indican la pertenencia a las otras categorías.

Por ilustrar el ejemplo de manera más sencilla, si tuviéramos una variable como el sexo (masculino/femenino), el one-hot encoding generaría dos columnas: una para masculino y otra para femenino. Sin embargo, aplicando drop_first=True, se obtiene una única columna que representa la presencia o ausencia de una de las categorías. En este caso, si se mantiene la categoría “femenino”, el valor 1 indicaría femenino y el valor 0 masculino. De este modo, se pasa de tener dos columnas a una sola, evitando redundancias y posibles problemas de multicolinealidad.

Además, este procedimiento permite convertir variables categóricas a formato numérico sin introducir un orden artificial entre categorías, lo cual es fundamental para el correcto funcionamiento de los modelos. En este ejercicio, el uso de one-hot encoding permitirá que los modelos de machine learning puedan interpretar correctamente variables como el mes, el tipo de visitante o el tipo de tráfico, evitando que se establezcan relaciones erróneas entre categorías y mejorando así la capacidad predictiva del modelo.

4.3 Escalado de variables

El escalado de variables consiste en transformar las variables numéricas para que todas tengan una escala comparable. Esto es importante porque algunas variables pueden tener valores mucho más altos que otras (por ejemplo, una variable con valores en torno a 3000 frente a otra entre 0 y 1), lo que podría hacer que el modelo les dé más peso del que realmente tienen.

En este caso, se aplicará un escalado de tipo StandardScaler, que transforma las variables para que tengan media 0 y desviación estándar 1. Este proceso es especialmente relevante para modelos como la regresión logística y las redes neuronales, ya que son sensibles a la magnitud de las variables.

Por el contrario, modelos basados en árboles como Random Forest no requieren escalado, aunque se aplicará igualmente para mantener coherencia en el preprocesado.

5.2 Red neuronal (MLPClassifier)

Se entrena un modelo de red neuronal multicapa (MLPClassifier) como alternativa más flexible a la regresión logística. Este tipo de modelo permite capturar relaciones no lineales entre variables, lo que puede resultar útil en problemas de comportamiento de usuario.

Para su entrenamiento, se utilizan los datos previamente escalados, ya que este tipo de modelos son sensibles a la magnitud de las variables.

Bloque 6 - Selección del umbral óptimo

Una vez seleccionado el mejor modelo, se procede a ajustar el umbral de decisión con el objetivo de optimizar la clasificación, en lugar de utilizar el valor por defecto de 0.5.

El umbral de 0.5 es un valor por defecto que no necesariamente es óptimo, especialmente en contextos de desbalanceo de clases como el presente. En este caso, utilizar dicho umbral podría llevar a una mala detección de las sesiones con compra.

Por ello, se recurre al análisis de la curva ROC, que permite evaluar el comportamiento del modelo a distintos umbrales y seleccionar aquel que optimiza la capacidad de discriminación entre clases.

In [20]:
from sklearn.metrics import roc_curve

# Obtener FPR, TPR y thresholds
fpr, tpr, thresholds = roc_curve(y_test, y_pred_proba)

# Encontrar el mejor umbral (máxima diferencia TPR - FPR)
optimal_idx = (tpr - fpr).argmax()
optimal_threshold = thresholds[optimal_idx]

print("Umbral óptimo:", optimal_threshold)

Umbral óptimo: 0.18
Una vez seleccionado el modelo, se procede a ajustar el umbral de decisión, ya que el valor por defecto de 0.5 puede no ser adecuado en contextos de desbalanceo de clases.

Mediante el análisis de la curva ROC, se determina un umbral óptimo de aproximadamente 0.18, que maximiza la diferencia entre la tasa de verdaderos positivos (TPR) y la tasa de falsos positivos (FPR), en otras palabras, con 0.5 solo se detctan usuarios muy seguros; mientras que con 0.18 ha capacidad de detectar más potenciales compradores.

Este resultado indica que el modelo es capaz de identificar potenciales compradores con probabilidades relativamente bajas, lo que puede ser útil en contextos donde se prioriza la detección de clientes con alta probabilidad de conversión, aunque esto implique aceptar un mayor número de falsos positivos.

Limitaciones del modelo:

  1. Una limitación relevante del modelo es que el dataset está compuesto por sesiones individuales y no por usuarios únicos. Esto implica que un mismo usuario puede aparecer en múltiples registros si ha accedido a la web en distintas ocasiones. Por tanto, los resultados obtenidos hacen referencia a sesiones con alta probabilidad de conversión, y no necesariamente a usuarios únicos. Sería recomendable contar con identificadores de usuario que permitan agrupar el comportamiento y mejorar la precisión del análisis.

  2. Posible data leakage (PageValues): algunas variables, como PageValues, pueden estar relacionadas directamente con la conversión, lo que podría inflar artificialmente el rendimiento del modelo si no se controlan adecuadamente.

  3. Falta de contexto del usuario: el modelo se basa únicamente en el comportamiento dentro de una sesión, sin considerar información adicional como historial de compras, perfil del cliente o variables externas, que podrían mejorar significativamente la capacidad predictiva.
